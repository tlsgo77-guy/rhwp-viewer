# 수행계획서: HWPX 저장 WASM API 노출 및 사용자 오류 전달

- **타스크**: [#176](https://github.com/edwardkim/rhwp/issues/176)
- **마일스톤**: M100
- **브랜치**: `local/task176`
- **작성일**: 2026-04-17

## 1. 배경

PR #170에서 HWPX Serializer 내부 모듈(`serialize_hwpx`)이 구현되었으나, WASM/JS에 노출되지 않아 웹뷰어에서 HWPX 저장이 불가능하다. 현재 `SaveAs()`는 항상 `exportHwp()`(HWP 바이너리)를 호출하여, HWPX로 열어도 HWP로 저장된다.

### 현재 문제 (#178)
1. HWPX 파일을 열고 편집 후 저장 → HWP 바이너리로 저장됨
2. 재저장된 파일을 다시 열면 렌더링 실패, 오류 메시지 없음
3. 사용자 기대: HWPX로 열었으면 HWPX로 저장

## 2. 목표

- `wasm_api.rs`에 `exportHwpx()` WASM 바인딩 추가
- `hwpctl/index.ts`의 `SaveAs()`에서 원본 파일 형식에 따라 HWP/HWPX 분기
- 저장 실패 시 사용자에게 오류 알림

## 3. 구현 단계 (3단계)

### 1단계: WASM API에 exportHwpx() 추가
- `wasm_api.rs`에 `exportHwpx()` 함수 추가
- `serialize_hwpx` 호출 → `Vec<u8>` 반환
- `SerializeError` → `JsValue` 변환 시 사용자 친화적 메시지

### 2단계: SaveAs() 파일 형식 분기
- 원본 파일 형식(HWP/HWPX) 추적 (`document_core` 또는 `wasm_api`)
- `hwpctl/index.ts`의 `SaveAs()`에서 형식에 따라 `exportHwp()` / `exportHwpx()` 분기
- 확장자를 원본 형식에 맞게 지정

### 3단계: 오류 전달 및 테스트
- 저장 실패 시 console.error + 사용자 알림
- WASM 빌드 후 웹뷰어에서 HWPX 저장 검증

## 4. 검증 방법

- `cargo test` 전체 통과
- WASM 빌드 후 웹뷰어에서 HWPX 파일 열기 → 편집 → 저장 → 다시 열기
- 저장된 파일이 HWPX(ZIP) 형식인지 확인
