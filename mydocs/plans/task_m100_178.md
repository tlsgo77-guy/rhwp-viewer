## 1. 배경

### 첫 시도 (4587685, 폐기) 의 잘못된 가정

UI 분기만 변경하여 HWPX 출처도 `exportHwp()` 호출 → HWP serializer 가 안정적이므로 한컴 호환 가능할 것이라는 가정.

### 실측 결과 (실패)

| 시나리오 | rhwp page_count |
|---|---|
| HWP 원본 → HWP 저장 → 재로드 | 1 → 1 (정상) |
| HWPX-h-01 → HWP 저장 → 재로드 | 9 → **209** (23배 폭주) |
| HWPX-02 → HWP 저장 → 재로드 | 6 → **155** (26배 폭주) |
| reflow_linesegs_on_demand 강제 후 | 효과 없음 |

### 진짜 원인 (작업지시자 통찰)

> "hwpx 의 경우 저장할 때는 hwp 시리얼라이제이션에 대한 매핑이 필요한건 당연한 것 아닌가?"

- **HWPX 파서**: vpos=0, `raw_ctrl_data` 비움, `attr=0`, `Section.raw_stream=None`, `table.common` 사용
- **HWP serializer**: `raw_ctrl_data` 우선, `attr` 비트 연산, `raw_stream` 우선, `table.attr` 사용

→ **HWPX-IR 과 HWP-IR 사이의 매핑 어댑터 부재**가 근본 원인.

### 작업지시자 결정

- 이슈 #178 컨텍스트에서 진행 (신규 이슈 X)
- **배포 일정 늦더라도 정도(正道) 진행**: HWPX→HWP IR 매핑 어댑터 본격 구현 후 안전한 저장 보장
- 사용자 영향: 매핑 어댑터 완성 전까지 HWPX 저장은 기존 깨진 동작 유지 (배포 보류 또는 현재 베타 유지)

## 2. 목표

**HWPX 원본을 rhwp 로 열고 편집 후 저장했을 때, 한컴2020에서 정상적으로 다시 열리는 HWP 파일이 생성**되도록 한다.

### 성공 기준

1. `samples/hwpx/hwpx-h-01.hwpx` 로드 → HWP 저장 → 한컴2020 정상 오픈
2. `samples/hwpx/hwpx-h-02.hwpx`, `hwpx-h-03.hwpx` 동일 통과
3. rhwp 가 자기 저장본 재로드 시 페이지 수 = 원본 (폭주 없음)
4. 편집 후 저장 시나리오도 같은 보장
5. HWP→HWP 라운드트립 회귀 없음
6. 단위 테스트 + 통합 테스트 그린

## 3. 통합 전략 원칙

1. **IR 정규화 패스 신설**: HWPX 출처 IR 을 HWP 의미로 정규화하는 별도 패스 (`hwpx_to_hwp_adapter`)
2. **저수준 serializer 변경 최소**: 기존 `serializer::cfb_writer` 는 안정적이므로 손대지 않음. 어댑터가 IR 을 HWP 친화 형태로 만들고 그대로 직렬화
3. **단계 분할 + 격리 검증**: 각 영역(페이지/표/셀/lineseg) 매핑마다 독립 테스트 + 페이지 수 회복 측정
4. **자동 차이 검출**: HWPX 출처 IR 과 HWP 출처 IR 의 필드 단위 비교 도구 신설
5. **재현 케이스 우선**: 큰 샘플 전에 가장 작은 샘플(ref_text)에서 페이지 폭주 재현 → 한 영역씩 매핑 적용 → 페이지 수 회복 측정
6. **단일 진입점**: `serialize_hwp` 호출 전 `convert_hwpx_ir_to_hwp_ir(&mut Document)` 자동 호출 또는 명시적 호출
7. **이중 안전장치**: 매핑 후에도 자기 재로드 검증 → 실패 시 사용자 알림 (조용히 손상된 파일 방치 금지, Discussion #188 원칙)

## 4. 범위

### 포함

- `src/document_core/converters/hwpx_to_hwp.rs` (가칭) 신규 모듈
- `convert_hwpx_to_hwp_ir(&mut Document)` 진입점
- 매핑 영역 (Stage 분할):
  - 페이지 컨트롤 (SectionDef, Section.raw_stream 합성)
  - 문단 break_type
  - 표 attr / raw_ctrl_data 재구성
  - 셀 list_attr / apply_inner_margin
  - lineseg vpos 사전계산
- 진단 도구:
  - HWPX vs HWP IR 차이 자동 추출 함수
  - 페이지 폭주 측정 회귀 테스트
- 단위 테스트 + 통합 테스트
- 사용자 흐름 연결: `file:save` 또는 `export_hwp_native` 에서 자동 또는 명시 호출

### 범위 외 (별도 이슈로)

- 도형 (Shape) 매핑 — Stage 5 또는 후속
- 각주/미주 매핑 — HWPX 파서 자체가 미구현
- 머리말/꼬리말 매핑 — HWPX 파서 자체가 미구현
- HWP→HWPX 변환 — 본 타스크 역방향 (별도 이슈)
- 모달 UI / 사용자 알림 보강 — 어댑터 안정화 후 후속 패치

## 5. 구현 단계 (6단계 예상)

본 수행계획서는 범위·접근 방향만 확정한다. **구체적 단계별 파일 변경 계획은 구현계획서(`task_m100_178_impl.md`) 에서 확정**한다.

단계 윤곽:

### Stage 1 — 진단 인프라 + 차이 자동 추출

- HWPX 로드 → IR 캡처 (deep clone)
- 같은 콘텐츠의 HWP 가 있다면 비교 (한컴 변환본). 없으면 HWP serializer 의 가정 영역 추출
- 차이 영역 자동 검출: section.raw_stream 유무, table.raw_ctrl_data 유무, table.attr vs common, 등
- 페이지 폭주 측정 회귀 테스트 (3개 샘플)

완료 기준: hwpx-h-01/02/03 의 차이 영역 목록 산출, 측정 베이스라인 (현재 폭주 배수) 기록.

### Stage 2 — 페이지 컨트롤 + Section.raw_stream 합성

- HWPX 출처의 `Section.raw_stream = None` 인 경우 HWP serializer 가 사용하는 stream 합성
- SectionDef → SECTION_DEF 컨트롤 + raw_ctrl_data 재구성
- 검증: ref_text.hwpx (가장 작은 샘플) 페이지 수 회복 확인

완료 기준: ref_text.hwpx → HWP → 재로드 페이지 수 = 1 (원본).

### Stage 3 — 문단 break_type + lineseg vpos

- HWPX `column_type` (ColumnBreakType) → HWP PARA_HEADER break_type 바이트
- lineseg vpos 사전계산: `reflow_line_segs` + `paginate` 결과로부터 추출
- IR 의 line_segs[].vertical_pos 갱신 (HWPX 의 항상 0 → HWP 절대 좌표)

완료 기준: hwpx-02 → HWP → 재로드 페이지 수 = 6 (원본).

### Stage 4 — 표 attr / raw_ctrl_data 재구성

- table.common (CommonObjAttr) → table.attr 비트 연산
- table.raw_ctrl_data 재구성 (위치·크기·여백 → 42+ 바이트 직렬화)
- cell.apply_inner_margin → list_attr bit 16

완료 기준: 표 포함 HWPX (예: ref_table 또는 hwpx-h-01) → HWP → 재로드 페이지 수 보존 + 표 정상 렌더.

### Stage 5 — 통합 검증 + serializer 자동 연동

- `serialize_hwp` 호출 전 자동으로 어댑터 호출 (또는 `export_hwp_native` 에서 source_format 검사 후 분기)
- 자기 재로드 검증 (저장 후 페이지 수 일치 확인) → 실패 시 사용자 알림
- 작업지시자 수동 검증 (hwpx-h-01/02/03 → 한컴 호환)
- 회귀: HWP→HWP 라운드트립 그린 유지

완료 기준: 3개 디버그 샘플 모두 한컴에서 정상 오픈, rhwp 자기 재로드 페이지 수 보존.

### Stage 6 — 사용자 경험 복원 + 배포 준비

- `file:save` / `hwpctl.SaveAs` 분기 변경 (#178 첫 시도 의 안전 부분 부활)
  - HWPX 출처도 HWP 저장 (이번엔 어댑터로 안전)
  - 파일명 정규화
- 단계별 보고서 + 최종 결과 보고서
- WASM 빌드 + 작업지시자 수동 검증
- 크롬 익스텐션 패키징 가능 상태

완료 기준: 사용자 검증 통과, 배포 패키지 준비.

## 6. 위험 요소

| 위험 | 완화 |
|---|---|
| 매핑 영역이 예상보다 광범위 | Stage 분할로 점진 진행. 단계마다 페이지 수 회복으로 진척 측정 |
| HWPX 파서가 채우지 않는 IR 필드 (예: FileHeader, version) | HWP 기본값 채우기. 향후 표준화 |
| 표·도형 raw_ctrl_data 재구성 실패 | 기존 HWP 샘플의 raw_ctrl_data 분석 + 역공학. 기존 hwplib 참조 |
| 단계간 회귀 (Stage 4가 Stage 2를 깨뜨림) | 누적 회귀 테스트 + 페이지 수 측정 자동화 |
| HWP→HWP 경로 회귀 | source_format 검사로 어댑터는 HWPX 출처에만 적용 |
| 매핑 후에도 한컴 거부 | 자기 재로드 검증 → 실패 시 알림. 매핑 추가 영역 발견·반복 |

## 7. 검증 방법

- **단위 테스트**: 각 매핑 함수의 IR 변환 결과 단위 검증
- **통합 테스트**:
  - `task178_hwpx_to_hwp_page_count_preserved` (각 단계마다 누적)
  - 3개 디버그 샘플별 회귀 테스트
- **자기 재로드**: HWPX → 어댑터 → HWP → DocumentCore::from_bytes 페이지 수 일치
- **수동 (작업지시자)**: 한컴2020 에서 저장 파일 정상 오픈
